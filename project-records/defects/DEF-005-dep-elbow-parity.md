## 事象

CR-003 Part 3 / DEP-L1-003 は依存線を「先行アイテムの右辺から出て後続アイテムの左辺へ入る」決定的
直交配線とし、DEP-L2-002 はその折れ点数が 0〜3 の範囲に収まると規定する。IM4 で実装した
`routeConnector`（右辺中央出／左辺中央入・横スタブ=矢じり先端の2倍・後続下=出直後下折れ・後続上=
後続直前上折れ・水平重なり時は行間ギャップを通過し両バーを跨がない）は、**前進依存**（後続が先行
より右に位置するケース。上下不整列=0折れ、上下に整列だが未重複=2折れ）では 0〜3 の制約を満たす。

しかし、**後続アイテムが先行アイテムより後方（左）に位置し、かつ両者が垂直方向に積み重なって水平
方向に重なっている**エッジケースでは、右辺出／左辺入という始点/終点アンカーのパリティ制約上、
行間ギャップを両バー非跨ぎで通過する非交差経路の折れ点数が **4** となり、DEP-L2-002 が明記する
「0〜3」の上限を超える（`tests/dependency-connector.test.ts` の該当ケースで検証・確認済み）。

## 根本原因

- 右辺出／左辺入という始点/終点アンカーの固定（DEP-L2-001 の specialize）は前進依存を主対象に
  設計されており（CR-003 Rationale「前進（後続が右）依存を主対象とし、遡り依存の詳細規則は本 CR
  では前進前提（将来 CR で拡張）」と明記）、後方ターゲットへの経路は考慮対象外だった。
- 後方＋垂直重複というエッジケースで両バーを跨がず非交差の経路を構成するには、右辺から一度離脱→
  垂直移動→行間ギャップ水平移動→再度垂直移動→左辺侵入、という最低 4 折れが幾何学的に必要
  （パリティ上、右辺出/左辺入を維持したまま 3 折れ以下で両バー非跨ぎの経路は構成できない）。
- IM4 実装ではこのエッジケースを「決定的・非交差」を優先する推奨仕様判断として 4 折れルートを
  正として実装し、DEP-L2-002 の数値上限からの逸脱として次セッションへ申し送りした。

## 影響

- 前進依存（主対象、CR-003 の主要受入基準）は全て規格内（0〜3折れ）。実運用上の大多数のケースは
  影響なし。
- 後方＋垂直重複という限定的なエッジケースのみ、DEP-L2-002 の記述文言（0〜3）と実装（4）が不一致。
  ユーザー視認上は経路が破綻・交差することはなく、決定的かつ両バー非跨ぎの経路として機能する
  （`tests/dependency-connector.test.ts` で非交差を検証）。
- 品質ゲート（Critical/High=0）はブロックしない。仕様文言と実装の数値不一致という文書整合性の
  問題であり、方針判断（仕様を緩和するか実装を変えるか）はユーザー判断が必要。

## 対応方針（ユーザー判断待ち、将来 CR）

以下のいずれかを将来 CR でユーザーに選択してもらう:

1. **DEP-L2-002 を緩和**: 「前進依存は 0〜3、水平重なり＋上下積みの後方ターゲットは最大4」へ
   規定を改訂し、実装（4折れ非交差ルート）をそのまま正とする。
2. **重なり時は上下端入線に変更**: 後方＋垂直重複ケースに限り、左辺中央入ではなく上端/下端から
   侵入する経路へ変更し、3 折れ以下に収める（DEP-L1-003/DEP-L2-001 の始点/終点アンカー規則の
   例外を追加する設計変更）。

いずれもコアドメイン「依存線の自動配線」に関わる設計変更のため、CLAUDE.md の重要判断基準
（アーキテクチャの根本的選択に準ずる整合性判断）に従い、change-manager 経由でユーザー確認の上
CR を起票して対応する。

## 対応（採用解） — 2026-07-21

上記方針案のうち **案 2（上下端入線）を採用**し、`routeConnector` を経路の相対配置ごとに
クリーンな 3 種へ再設計した（左辺中央入の固定を撤廃し、重なり/後方時は上下端入線へ切替）:

1. **CLEAR FORWARD**（後続の左辺が先行の右辺よりスタブ 1 本以上右）: 従来どおり右辺中央出→左辺
   中央入（整列=0折れ / 下=出直後下折れ2 / 上=後続直前上折れ2）。**回帰なし**。
2. **CONTIGUOUS 同一行 FS**（隣接して同レーンに並ぶ。後続左辺≒先行右辺で左辺入が退化＝
   ゼロ縦幅ナブ）: 行の下へスタブ 1 本分落とし、後続の少し内側の列まで走り、後続の **下端** へ
   立ち上げる可視の「コ」の字（2 折れ、縦幅 > 0、フラットナブ廃止）。
3. **STACKED / OVERLAPPING・後方**（別レーン。後続左辺が先行右辺の左＝左辺入だと先行を跨いで
   左へ遡る）: 右へ出て後続上の列（後続が下なら上端、上なら下端の facing edge に対応）へ走り、
   **上端/下端** から差し込むクリーンな L（1 折れ）。**先行の出口 x より左へは一切進まない**。

いずれも折れ点 **0〜3** に収まり、両バー・自バーを跨がない。従来 4 折れを強いていた「右出/左入」
のパリティトラップは、重なり/後方時に上下端入線へ切替えることで解消。**DEP-L2-002（0〜3）を
満たすため仕様緩和は不要**（案 1 は不採用）。

デフォルトテンプレートの 4 依存（いずれも同一マルチバー行内）の新経路:

- `tpl-dep-concept-dev` / `tpl-dep-dev-valid`（接触・同レーン・同 y）: 下端入線の可視「コ」の字
  （出口 → 下スタブ → 内側列へ横移動 → 後続下端へ立上げ）。従来の ~12px フラットナブを解消。
- `tpl-dep-sys1-sys2` / `tpl-dep-sys3-swe1`（時間重なり・別レーン。後続左辺が先行右辺より左だが
  後続右辺は先行右辺より右）: 上端（または下端）入線の L（出口 → 右へ短く → 後続 facing edge へ
  縦入り）。従来の「左へ遡るジグザグ/ループ」を解消し、先行出口 x より左へ進まない。

## 再発防止（プロセス改善提案）

  時に数値制約の網羅的な再検証（境界ケース含む）をレビューの必須観点に追加することを検討する。

## 再オープン（CR-008, 2026-07-21）

- 関連CR: **CR-008**（`change-request-008-20260721-070348.md`）。IM8 の上下端入線方針を上書きする。
- 再オープン理由: IM8 で採用した「上下端入線（水平スタブ無し）」方式は、実データ（実プロジェクトの
  サンプル画像）レビューで以下 2 点の視認性不具合が確認され、CR-008 が新たに確定した「出口/入口の
  水平スタブ必須」および「視認方向の正しさ」要求を満たさない:
  1. **SYS1 → SYS2（同一行・時間重なり）**: 後続の上端へ縦入線するが矢印手前に水平部が無いため、
     線が垂直のまま矢じりへ突入し、依存の終端が読み取りにくい（vertical-into-arrow）。
  2. **Concept → Series Development（同一行連続・隣接）**: U 字ルートを取るため、一見すると依存の
     向きが逆（後続から先行へ向かうよう）に見える（reversed-looking U）。
- 再解決方針: CR-008 の実装により再解決する。すなわち、
  (a) 先行の出口直後と後続の入口直前の双方に `CONNECTOR_STUB_PX`（矢じり先端の 2 倍）の水平スタブを
      必ず挿入し、上下端入線でも入口直前を水平で終端させる、
  (b) 同一行連続ケースの逆向きに見える U 字を、依存の向きが正しく読める経路へ変更する、
  (c) 折れ点上限を「前進（後続が右かつ非隣接）= 0〜3、重なり/後方/同一行連続 = 最大 4」へ緩和する
      （DEP-L2-002 を CR-008 で改訂済み）。
- 仕様同期: `docs/spec/16-dependencies.sdoc` の DEP-L1-003 / DEP-L2-001 / DEP-L2-002 を CR-008 の
  内容で改訂済み（2026-07-21）。実装（`src/domain/usecase/dependency-connector.ts` の `routeConnector`
  再設計）と対応テスト（`tests/dependency-connector.test.ts`）は実装セッションで一体反映する。
