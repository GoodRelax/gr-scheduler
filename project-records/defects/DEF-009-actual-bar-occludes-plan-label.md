# DEF-009: 実績バーが予定アイテムの略称ラベルを覆い隠す

- 起票: 2026-07-22（ユーザーの実機ビジュアルレビュー）
- 種別: defect（描画層の重ね順。可読性の要求未充足）
- 重大度: **High**（一枚絵の中でアイテムを識別する主要手段である略称が読めなくなる。ITEM-L1-009 の
  「略称をアイテム内部/付近に表示できること」が実質的に満たされない）
- 状態: **Fixed（2026-07-22、実機ライブ検証済み）**
- 関連: ITEM-L1-009（略称表示）, ITEM-L2-002（タスクラベル既定位置 inner-left）,
  PLAN-L1-005（Overlap/Separate）, CR-012（サンプルを統一モデルへ移行したことで顕在化）

---

## 事象

Overlap（`[Ao]`、既定）表示で、実績を持つタスクの **略称ラベルが実績バーに塗り潰されて読めない**。

ユーザー提供のスクリーンショットで確認された例:
- `Concept` — 実績バーが予定バーの上に重なり、"Concept" の文字がほぼ隠れている。
- `Series Development` — 同様に文字が実績バーに覆われている。

実績を持たないアイテム（`Validation` / `SYS.4` 等）は正常に読める。したがって
**実績バーが描かれるアイテムに限って発生**する。

## 原因

`src/adapters/render/layers/item-layer.ts` は 1 アイテムのノードを

1. プライマリグリフ（＝予定バー。`inner-left` の略称テキストを伴う）
2. 実績バー（`updateActualBar`）

の順に生成・追加している。SVG は**後から追加した要素が前面**に描かれるため、実績バーが
予定バーとその略称テキストの上に乗る。ラベルは予定バーと同じ層に属しており、
実績バーより前面へ持ち上げる処理が無い。

Overlap は仕様上「予定バーの上に実績を重ねて塗る」（`plan-actual-geometry` の設計意図）ため、
重なること自体は正しい。**欠落しているのは「ラベルだけは常に最前面」という重ね順の規定と実装**。

## あるべき挙動

- 略称ラベル（および担当者名ラベル等のテキスト）は、予定バー・実績バーの**いずれよりも前面**に
  描画し、Overlap / Separate のどちらでも、実績の有無に関わらず可読であること。
- 色の意味（淡＝予定 / 濃＝実績、CR-002）は変更しない。半透明化などで配色の意味を薄めない。
- ユーザー確定方針（2026-07-22）: **「ラベルを最前面へ」** を採用する。

## 修正方針

`item-layer` のノード生成順（または `z` 相当の再配置）を見直し、テキスト要素を実績バーより後に
追加する／テキスト専用の前面グループへ移す。実績バーの生成・更新・削除
（`updateActualBar` / `removeActualBar`）を繰り返してもラベルが背面に落ちないこと。

## 検証

- 実績を持つタスク（`oa-phase-plan-concept` / `oa-phase-plan-dev` 等）で、Overlap・Separate の
  双方で略称が読めること（DOM 上でテキスト要素が実績バーより後方＝前面にあること）。
- `[P]`/`[A]` トグルを往復しても重ね順が壊れないこと。
- 単体テスト（レンダー経路）＋実機 DOM の両方で確認する。
