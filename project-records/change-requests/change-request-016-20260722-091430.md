## 1. 背景・目的

本セッションでユーザーがヘッダー中央のプロジェクト名表示を見て、「プロジェクトのタイトルを編集
できない」「出力用の JSON/XML にもプロジェクト名を出力すること」と指摘した。調査の結果、次の
事実が判明した。

### 出力（JSON/MSPDI）は既に実装済みである

- `docs/api/gr-scheduler.schema.json` は `title` を必須フィールドとして定義している。
- `src/domain/usecase/json-codec.ts` は `title` を必須文字列として検証・入出力しており、JSON へは
  **既に往復している**。
- `src/domain/usecase/mspdi-codec.ts` は Export 時に `<Project><Title>...</Title></Project>` を
  出力し、Import 時に `<Title>` を読む（既定値 `'Imported project'`）。MSPDI へも**既に往復して
  いる**（`docs/spec/40-data-format.sdoc` の `DATA-MSPDI-001` が `title -> Project/Title` の
  マッピングを規定）。

すなわち「出力されない」というユーザーの当初理解は不正確であり、真の問題は別にある。

### ヘッダー表示のモデル連動状況（DEF-010、本 CR 起票時点での読取専用確認）

DEF-010（`project-records/defects/DEF-010-header-title-hardcoded.md`、2026-07-22 起票）は、ヘッダー
のプロジェクト名表示が `scheduleNameLabel.textContent = 'gr-scheduler';` というリテラル固定で
`ScheduleDocument.title` を一切参照していない、と記録している。

本 CR の起票にあたり `src/app/main.ts` を読取専用で再確認したところ、次の点で DEF-010 の記述と
現行ソースにずれがあることが判明した。

- `syncScheduleName(chrome, store)` という関数が存在し、`store.getDocument().title` を読んで
  `chrome.scheduleNameLabel.textContent` へ反映している（空文字の場合のみ `'gr-scheduler'` へ
  フォールバック）。
- この関数は、(a) アプリ初期化時に 1 回（`main.ts` 内、セットアップ処理の一環）、および (b) JSON/
  MSPDI の Import 成功時（同等のロジックがインライン展開された箇所）の**2 箇所**で呼ばれている。

すなわち、DEF-010 が記録した「モデルへの参照が一切無い」という状態は、少なくとも初期表示と
明示的な Import 操作については既に解消されている可能性がある（本 CR 起票時点のソース確認による
所見であり、DEF-010 自体の記述更新・再判定は defect 管理側で改めて行うべき事項として記録する。
下記 §7 参照）。

一方で、次の**真の欠落**が残っている。

1. **タイトルを編集する UI が一切存在しない。** `syncScheduleName` は表示の同期のみを行い、
   ユーザーがタイトルを変更する手段（本 CR Part 1 が新設する機能）は現行どこにも無い。
2. **`syncScheduleName` は「初期化時」と「Import 時」の 2 つの時点でのみ個別に呼ばれており、
   ストア／ドキュメント変更に対する継続的な購読（subscribe）として組み込まれていない。** すなわち
   Import 以外の経路でタイトルが変わった場合（本 CR Part 1 のダブルクリック編集や、将来の Undo/
   Redo によるタイトル変更）に、ヘッダー表示が自動的に追従する保証が無い。本 CR Part 1 の実装は、
   この同期をどう組み込むか（`syncScheduleName` を汎用の描画/購読フックへ統合するか、編集確定時に
   明示的に呼び出すか）を実装フェーズで確定する必要がある。
3. **既定テンプレートの `title`（`'gr-scheduler'` または `'gr-scheduler template'`、
   `src/app/sample-data.ts` 参照）が、ヘッダーの空文字フォールバック文字列 `'gr-scheduler'` と
   偶然一致・酷似しており**、ユーザーが実機で見た際に「タイトルが表示されているのか、単なる既定
   文字列なのか」を区別できず、これが「タイトルが出力されない（連動していない）」という当初の
   印象の一因になった可能性がある。

本 CR は、上記のうち **1（編集 UI の新設）** を Part 1 として、**MSPDI `<Name>` 出力の追加** を
Part 3 として新規要求で扱う。**2（継続的な同期の欠落）** は Part 1 の実装が正しく機能するための
前提条件として実装フェーズで解消することを明記する（DEF-010 と一体で修正）。**3（既定文字列の
偶然の一致）** は本 CR のスコープ外の観察事項として記録するに留める。

## 2. 変更内容

### Part 1 — プロジェクトタイトルのダブルクリックによるインライン編集

- ヘッダーに表示されているプロジェクトタイトルを、**ダブルクリックでテキスト編集モードに入り、
  Enter で確定・Escape で取消**できるようにする。操作系は CR-007 Part 3（コメントのダブルクリック
  編集、Enter で確定）に倣う。CR-007 Part 3 は起票時点で Escape の取消要否を「実装フェーズで確認」
  としていたが、本 CR ではタイトル編集について **Escape で編集前の値へ復元する（取消する）** ことを
  明示的に確定する。
- 編集の確定（Enter またはフォーカス外れによるコミット等、具体的なコミット契機は実装フェーズで
  CR-007 Part 3 の実装と整合させて確定する）は、`ScheduleDocument.title` を更新するコマンドとして
  発行し、**Undo 可能**とする（TOOL-L1-004、既存の Undo/Redo 機構に統合する）。
- 編集確定後、ヘッダー表示（`scheduleNameLabel`）が即座に新しいタイトルを反映すること。上記 §1 の
  「継続的な同期の欠落」を、本 Part の実装と一体で解消する（DEF-010 の修正方針「ストア／レンダラの
  ドキュメント変更通知に購読して title を反映する」を実装する）。

### Part 2 — DEF-010（ヘッダーとモデルの表示連動）の記録更新

本 CR の起票調査により、DEF-010 が記述する「リテラル固定・モデル未参照」という状態は、少なくとも
初期表示・Import 時点については `syncScheduleName()` により既に部分的に解消されていることが判明
した（上記 §1 参照）。**残っている真の欠落は「編集 UI が無いこと」（Part 1 で新設）と「継続的な
同期が無いこと」（Part 1 の実装で一体解消）である。** DEF-010 の記録内容（事実確認節）は、本 CR の
実装時に最新のソース状況へ更新するか、あるいは「編集 UI 新設に伴い解消」として Closed 化するかを
defect 管理側で判断する（下記 §7 未決事項）。

### Part 3 — MSPDI `<Name>` への出力追加

- `src/domain/usecase/mspdi-codec.ts` の Export 処理に、`<Title>` に加えて **`<Name>` へも
  プロジェクト名を出力する** 処理を追加する（`docs/spec/vendor/mspdi/mspdi_pj12.xsd` は `Project`
  要素の子として `Name`（248行目）と `Title`（258行目）をともに任意要素として定義しており、
  双方へ出力することで MS Project がプロジェクト名として認識する箇所（`Name`）にも値が現れるよう
  にする）。
- Import 時は **`Title` を正とし（authoritative）、`Title` が存在しない場合に限り `Name` へ
  フォールバックする**（現行の Import ロジック `firstTagText(xmlText, 'Title') ?? 'Imported
  project'` を、`firstTagText(xmlText, 'Title') ?? firstTagText(xmlText, 'Name') ?? 'Imported
  project'` 相当へ拡張する）。
- `docs/spec/40-data-format.sdoc` の `DATA-MSPDI-001` を、`title -> Project/Title`（Export の主・
  Import の正）に加え `title -> Project/Name`（Export の副次出力・Import のフォールバック）を
  規定するよう改訂する。

## 6. 受入基準（Acceptance Criteria）

- [ ] ヘッダーのプロジェクトタイトルをダブルクリックすると編集モードに入ること。
- [ ] 編集中に Enter を押すと入力内容が `ScheduleDocument.title` へ確定され、ヘッダー表示が即座に
      新しいタイトルへ更新されること。
- [ ] 編集中に Escape を押すと、入力内容が破棄され編集前の表示へ復元されること（`title` は変更
      されないこと）。
- [ ] タイトル変更が Undo 可能であること（Undo で編集前のタイトルへ戻り、Redo で編集後のタイトルへ
      戻ること）。
- [ ] JSON の Load/Save で `title` が既存通り往復すること（回帰確認）。
- [ ] MSPDI Export で `<Project>` 要素に `<Title>` と `<Name>` の両方が出力され、いずれも同じ
      プロジェクト名を保持すること。
- [ ] MSPDI Import で、`<Title>` が存在する場合はその値が優先されること。`<Title>` が存在せず
      `<Name>` のみ存在する場合は `<Name>` の値がタイトルとして採用されること。両方とも存在しない
      場合は既定値（`'Imported project'`）が使われること。
- [ ] 既定テンプレート起動時、ヘッダーにテンプレートの実際の `title` が表示されること（既定の
      フォールバック文字列との区別が実装/テストの両面で確認できること）。
- [ ] 品質ゲート（tsc / vitest / eslint / strictdoc / build）が引き続き green であること。

## 7. 決定事項・未決事項

  具体的な結線方法（既存の描画ループへ統合するか、タイトル編集コマンドの確定時に明示的に呼び出す
  専用の結線とするかを含む）。
- タイトル編集のコミット契機の詳細（Enter のみか、フォーカス外れ・他要素クリックでもコミットする

- 既定テンプレートの `title`（`'gr-scheduler'`/`'gr-scheduler template'`）が空文字フォールバック
  文字列 `'gr-scheduler'` と酷似している点（上記 §1 の 3）について、既定テンプレートの `title` を

## 8. 段階化の注意（仕様先行・実装後追い）

CR-001 §8 / CR-011 §8 / CR-012 §8 / CR-013 §8 / CR-014 §8 / CR-015 §8 と同様の理由（`docs/api/
gr-scheduler.schema.json` は F0 で SSOT 化され `tests/document-schema-conformance.test.ts` がコード
出力と突合しているため、コードより先に schema を変更するとテストが赤化する。本 CR はスキーマ変更を
