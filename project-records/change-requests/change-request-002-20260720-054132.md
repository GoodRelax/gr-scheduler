## 1. 背景・目的

CR-001（実績日フィールド方式への移行、2026-07-19 承認）の仕様確定後、本セッションでの設計質疑を通じて、予実表示の
細部について 3 点の追加方針決定が生じた（Q1〜Q3）。いずれもユーザーが選択肢を比較検討した上で
確定した requirement-change であり、change-manager を経由して記録する。

1. **Q1 配色**: Overlap の予実塗り分けは、アイテムのベース色（fillColor）1 色から彩度で導出する
   （予定＝淡色・実績＝濃色）。非色の冗長符号（WCAG 1.4.1）は破線ではなく線の太さで表す。
2. **Q2 マイルストーン**: 幅 0 の点であるマイルストーンは、区間を塗る「予実バー」ではなく
   「予定マーカー＋実績マーカーの 2 点」で予実ズレを表す。イナズマ線 front はマイルストーンでは
   点に退化する特例が要る。
3. **Q3 ベースライン**: CR-001 Part A で「据え置き」とした `item.previousPlan` フィールドを
   **廃止**し、変更前予定（ベースライン）は別ファイル参照方式（過去予定スナップショット文書を
   グレー・編集不可のアンダーレイとして読み込む）へ変更する。per-item 入力の負担と、フィールド/
   ファイルの二重管理（SSOT違反）を避けるため。これは CR-001 Part A の一部および Part B-4 の
   MSPDI Baseline マッピングを **supersede** する。

本 CR は上記 3 点を Part 1〜3 として 1 パッケージにまとめ、CR-001 と同様に仕様（`.sdoc`）確定を
先行し、schema / `src/**` の反映は別セッション（実装フェーズ）へ引き継ぐ。

## 2. 変更内容

### Part 1 — 予実の配色（Overlap 塗り分け規則）

- ユーザーは任意のアイテム色（`fillColor`）を選べる（プロパティ/パレット）。**据え置き**。
- Overlap の予実塗り分けは、アイテムのベース色 1 色から**彩度**で導出する: **予定＝淡色
  （desaturated/pale）、実績・進捗＝濃色（saturated/vivid）**。既定はこの淡（予定）/濃（実績）。
- 非色の冗長符号（WCAG 1.4.1、色覚特性/グレースケール対策）は**破線を用いない**。代わりに
  **線の太さ**で区別する: 予定線は細く（補足的）、実績線は太く描く。破線は情報過多で見にくい
  というユーザー判断により不採用。
- 旧 `planActualKind` 由来の固定 緑(plan)/橙(actual) 2 色は**廃止**（`plan-actual-colors.ts` を
  彩度＋線幅方式へ作り替える＝実装フェーズ）。
- 明示 `fillColor`（`fillColorExplicit`）は従来通り優先。ラベル可読性（WCAG 1.4.3）は維持する。

### Part 2 — マイルストーンの予実描画

- マイルストーンは幅 0 の点（`milestoneShape`）。予実は「マーカー 2 点」＝**予定マーカー
  （`startDate`）＋実績マーカー（`actualStart`）**で表す。両者の**間は塗りつぶさない**（塗る
  区間が無い）。ズレは 2 点の水平距離＋イナズマ線頂点で表す。任意で 2 点を結ぶ細いリーダー線で
  シフトを示すのは可。
- milestone は `actualEnd = null`（**据え置き**、CR-001 Part A のまま）。
- イナズマ線 front（PLAN-L2-001）はスパン前提の 4 ケース補間だが、マイルストーンは区間が
  無く退化するため特記する: **マイルストーンの front は「点」＝実績あれば `actualStart`、
  無ければ `startDate`。補間しない。** PLAN-L2-001 にマイルストーン特例を追加する。

### Part 3 — 変更前予定（ベースライン）を「別ファイル参照方式」へ変更（CR-001 supersede）

- **`item.previousPlan` フィールドを廃止する**（CR-001 Part A が「据え置き」とした点を
  **supersede**）。
- ベースライン（過去の公式予定）は**別ファイル**（過去予定スナップショット文書）として読み込み、
  **薄いグレー・編集不可のアンダーレイ**として描画する。その参照ファイル内の実績は無視する
  （予定日のみ使用）。
- 現行アイテムと参照ファイルのアイテムの対応付けは **`id` をキーに突合**する。
- MSPDI Baseline（`BaselineStart`/`BaselineFinish`）は、読み込んだ参照文書を `id` 突合して
  **best-effort で合成/往復**する（CR-001 B-4 の previousPlan ↔ Baseline の clean 往復は
  **best-effort へ格下げ**）。
- **意図**: 「実績が遅すぎて已む無くマイルストーン/予定を変更したが、現在はこれが正規・公式の
  日程」という履歴を語るための表示。per-item で毎回 `previousPlan` を入力する負担を避け、
  ベースラインの持ち方を「フィールドとファイルの 2 系統」にしない（SSOT・二重管理排除）。
- **描画高さ（ユーザー確定、2026-07-20）**: ベースライン（過去予定）は対象行と**同じ高さ**に
  薄いグレーで描く。高さは変えない（オフセットしない）。行内の上下位置・サイズは現行アイテムの
  レイアウト結果に一致させ、別トラック化やずらし表示は行わない。
- **参照形式（ユーザー確定、2026-07-20）**: ベースライン参照は**JSON のみ**（MSPDI XML は不可）。
  JSON を Load する際のオプションで「ベースライン扱い」であることを明示指定して読み込む
  （通常の編集対象としての Load とは区別する UI/API フラグを設ける）。
- **可視制御（ユーザー確定、2026-07-20）**: ベースライン層の表示/非表示は**専用トグル**
  （Visible / Invisible）で切り替える。このトグルは**ヘッダーに配置**する（ヘッダー配置の詳細は
  CR-003 に従う）。可視制御は予実の `planActualDisplay` とは**独立**とする。

## 6. 受入基準（Acceptance Criteria）

- [ ] Overlap が淡（予定）/濃（実績）の彩度導出で描かれ、非色冗長符号は線幅（予定細/実績太）で
      表されること（破線を用いない）。
- [ ] マイルストーンの予実が「予定マーカー＋実績マーカーの 2 点」・塗り無しで描かれ、イナズマ線
      front が点（補間しない、実績あれば `actualStart`・無ければ `startDate`）で算出されること。
- [ ] `item.previousPlan` フィールドが廃止され、変更前予定は別ファイル参照のグレー・編集不可
      アンダーレイとして `id` 突合で描かれること。
- [ ] MSPDI Baseline（`BaselineStart`/`BaselineFinish`）が参照文書の `id` 突合で best-effort
      往復すること。
- [ ] 単体テスト／往復整合テストが green であること（品質目標準拠）。

## 7. 決定事項・未決事項

**決定済み（本セッションでユーザー承認、2026-07-20）:**
- Q1 配色: ベース色 1 色からの彩度導出（淡=予定/濃=実績）、非色冗長符号は線幅（破線不採用）。
- Q2 マイルストーン: 予定/実績の 2 マーカー方式（区間を塗らない）、front は点に退化（補間なし）。
- Q3 ベースライン: 別ファイル参照方式へ変更、`previousPlan` フィールド廃止、`id` 突合、
  MSPDI Baseline は best-effort 往復へ格下げ。CR-001 Part A（previousPlan 据え置き）と
  Part B-4（Baseline clean 往復）を **supersede**。
- 描画高さ・参照形式・可視制御（追記、2026-07-20）: ベースラインは対象行と同じ高さで薄グレー
  描画（オフセットしない）。参照形式は JSON のみ（XML 不可）、Load 時のオプションで「ベースライン
  扱い」を明示指定。可視制御は専用トグル（Visible / Invisible）をヘッダーに配置し、
  `planActualDisplay` とは独立に扱う（ヘッダー配置詳細は CR-003 準拠）。

**未決事項（実装フェーズで確定）:**
- ヘッダー上のトグル UI の具体的な配置・意匠（CR-003 のヘッダー配置方式の確定を待つ）。
- ベースライン参照文書の JSON スキーマ上の扱い（gr-scheduler JSON のサブセットとして扱うか、
  専用のバリデーション規則を設けるか）の詳細。
