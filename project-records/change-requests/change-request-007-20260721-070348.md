## 1. 背景・目的

現行仕様/実装では、選択操作は矩形選択のみで PowerPoint 的な増減（Ctrl+クリック）ができず、複数
選択したアイテムの一括移動は「同一の日付シフト」しか想定されていない（分類をまたいだ再配置の規則
が未定義）。またコメント（CURS-L1-005）はテキストを保持できる旨は仕様化済みだが、編集を起動する
具体的なジェスチャ（ダブルクリック→編集、Enter→確定）が未規定であり、実装（`Ctrl+A` は
`src/adapters/input/keyboard-shortcuts.ts` の `selectAll()` で既に実装済み）はアイテムのみを対象
としコメントを含まない。さらに分類（大/中/小分類・セクション）を複製する UI が存在しない。

本セッションでユーザーと協議の上、複数選択の PowerPoint 式増減（Part 1）、複数アイテムの左右/
上下移動規則（Part 2、★2 確定）、コメント編集ジェスチャ（Part 3）、Ctrl+A へのコメント包含
（Part 4）、分類のコピー&ペースト UI（Part 5、★1 確定）を 1 パッケージの変更要求として確定した。
CR-001/002/003 と同様に仕様（`.sdoc`）確定を先行し、実装は別セッションへ引き継ぐ。

## 2. 変更内容

### Part 1 — 複数選択の拡張（Ctrl+クリックによる増減）

- 現状は矩形選択のみ。**Ctrl+クリックで個別アイテムの選択/選択解除の増減**を可能にする
  （PowerPoint 式）。矩形選択で選び過ぎた／選び足りなかった分を、続けて Ctrl+クリックで増減できる
  こと（矩形選択の結果セットへの追加・除外の両方）。
- 現行仕様には「矩形選択」自体および「選択の増減」を明文化した専用要求 UID が存在しない
  （ITEM-L1-001 は「各アイテムが独立に選択できること」に留まる）。本 CR は srs-writer による
  新規要求追加（`11-items-icons.sdoc` への選択セクション新設、STK-L0-001/STK-L0-008 を Parent と
  する L1/L2 要求）を実装フェーズ前提として求める。

### Part 2 — 複数アイテムの移動（★2、協議確定）

複数選択したアイテムを、ドラッグ方向によって次の 2 モードで移動する。

- **(a) 左右移動 = 日付の一括シフト**: 全選択アイテムの開始日・終了日を同一移動量だけシフトする
  （まとめて遅延/前倒し）。ALIGN-L1-002（ドラッグ移動）・ALIGN-L1-003（移動連動のプロパティ自動
  更新）の複数選択への拡張。
- **(b) 上下移動 = 選択が共有する最深の分類レベルでの隣接 sibling への再割当**: 規則は以下。
  - 選択アイテムが**大分類のみ**共有 → 隣接**セクション（major）**へ再割当。
  - **中分類まで**共有 → 同一 major 内の隣接**中分類（middle）**へ再割当。
  - **小分類まで**共有 → 同一 middle 内の隣接**小分類（minor）**へ再割当。
  - 上位レベルの分類は維持し、再割当対象レベルのみを変更する。分類ツリーの端（最初/最後の
    sibling）では停止し、それ以上は移動しない。
  - この規則により、上下移動は「所属分類（`majorCategory`/`middleCategory`/`minorCategory` および
    対応する `rowId`）の変更のみ」で完結し、階層構造そのもの（大中小の親子関係）は不変。ロジックが
    単純になる（アイテムの複製や新規行生成を伴わない）。

対象: ALIGN-L1-002/ALIGN-L2-001（整列制約ソルバへ「分類再割当」モードを追加）、SECT-L1-002
（複数行のセクション化・順序）、SECT-L2-001（大中小のインデント階層表現。上下移動の「深さ」判定は
本表現のツリー構造に基づく）、DATA-JSON-003/004/005（`sections[].rowIds[]`,
`rows[].majorLabel/middleLabel/minorLabel`, `items[].majorCategory/middleCategory/minorCategory`
の更新のみで表現可能。依存線は同一アイテム ID を保つため Part 5 のような複製時の依存線処理は不要）。

### Part 3 — コメントの編集

- 現状コメント（CURS-L1-005）はテキストを保持できるが、編集を起動する具体的ジェスチャが未規定
  であり実装も未対応。**コメントアイテムをダブルクリックでテキスト編集モードに入り、Enter で確定**
  できるようにする。
- CURS-L1-005（コメントのテキスト記載・移動・編集の L1 要求）・CURS-L2-002（アンカー+吹き出し
  オフセット+引出し線の位置モデル）に、編集起動ジェスチャ（dblclick→編集、Enter→確定、Escape での
  取消要否は実装フェーズで確認）を L2 要求として追記する。

### Part 4 — Ctrl+A でコメントも選択対象に含める

- 現状 `Ctrl+A` はアイテムのみを全選択する（`src/adapters/input/keyboard-shortcuts.ts` の
  `selectAll()` が `context.store.getDocument().items` のみを対象にしている）。**コメントも Ctrl+A
  の全選択対象に含める**よう拡張する。
- TOOL-L1-005（ショートカットキー）のキー割当表に `Ctrl+A` の行が存在しないため、本 CR で表に
  `Ctrl+A = 全選択（アイテム + コメント）` を追加する。選択モデル（Part 1 の増減も含む）がコメント
  を選択可能な対象として扱うことが前提となる。

### Part 5 — セクション/分類のコピー&ペースト（★1、協議確定）

クラシフィケーション（大分類・中分類）の複製 UI を新設する。

- **複製位置**: 選択中の分類レベルに応じて、**大分類選択時**＝当該大分類の直下（次の大分類の
  直前）に複製を挿入。**中分類選択時**＝当該中分類の直下（次の中分類の直前）に複製を挿入。
- **複製名**: 既存名 + 次の空き番号 suffix（例: `Body` → `Body-1`。既に `Body-1` が存在すれば
  `Body-2`）。
- **複製の中身**: 配下の行（rows）・アイテム（items）を**新規 ID**で複製し、`rowId` と分類
  （`majorCategory`/`middleCategory`/`minorCategory` および `sections[].rowIds[]`）を複製側へ
  張り替える。
- **依存線の扱い**: セクション/分類をまたぐ依存線（DEP-L1-001 の始点/終点が複製元の分類外にある
  もの）は、**複製側では落とす**（原本アイテムを指したまま残さない）。同一複製内で閉じる依存
  （複製された両端が同じ複製操作内にある場合）の扱いは実装フェーズで確定する（§7 未決事項）。
- **UI操作子**: 左ペインの分類ノードの並替（▲▼）近傍に、コピペアイコン（2 枚重ねの書類）を配置
  する。
- 対象: SECT-L1-002（セクション化と順序）、SECT-L1-006（分類名の左ペイン表示）、SECT-L2-001
  （大中小のインデント階層。複製 UI の配置場所の前提）、DATA-JSON-003/004/005（新規 ID 生成・
  `rowId`/分類の張替えの対象データ）、TOOL-L1-003（アイテムのコピー&ペーストの既存パターンを
  分類複製へ準用）。

## 6. 受入基準（Acceptance Criteria）

- [ ] 矩形選択の結果に対し、Ctrl+クリックで個別アイテムを追加選択・選択解除できること（矩形選択の
      過不足を Ctrl+クリックで補正できること）。
- [ ] 複数選択したアイテムを左右にドラッグすると、全選択アイテムの開始日・終了日が同一移動量だけ
      シフトすること。
- [ ] 複数選択したアイテムを上下にドラッグすると、選択が共有する最深の分類レベル（大/中/小）に
      応じて隣接 sibling（セクション/中分類/小分類）へ再割当され、上位レベルの分類は変化しない
      こと。分類ツリーの端（sibling がない）では移動が発生しないこと。
- [ ] コメントアイテムをダブルクリックするとテキスト編集モードに入り、Enter で編集内容が確定
      すること。
- [ ] `Ctrl+A` を押すとアイテムとコメントの両方が全選択されること。
- [ ] 左ペインの分類ノードの▲▼近傍にあるコピペアイコンから、選択中の分類レベル（大分類/中分類）
      に応じて当該分類の直下に複製が挿入されること。
- [ ] 複製名が「既存名 + 次の空き番号 suffix」（例: `xxx-1`。既存なら `xxx-2`）で採番されること。
- [ ] 複製配下の行・アイテムが新規 ID で複製され、`rowId`/分類が複製側に正しく張り替わること。
- [ ] 複製元の分類/セクションをまたぐ依存線が、複製側では生成されない（原本アイテムを指したまま
      残らない）こと。
- [ ] 上記すべての操作が Undo/Redo 可能であること。
- [ ] 単体テスト 95% 以上／結合テスト 100%（品質目標準拠）で green であること。

## 7. 決定事項・未決事項

**決定済み（本セッションでユーザー確定、2026-07-21）:**
- Part 2（★2）: 複数アイテムの上下移動は「選択が共有する最深の分類レベルでの隣接 sibling 再割当」
  であり、階層構造自体は不変（分類の付け替えのみ）。左右移動は日付の一括シフト。
- Part 5（★1）: 分類複製は「選択中の分類レベル直下への挿入」「既存名+次の空き番号 suffix」
  「配下の新 ID 複製＋分類またぎ依存線は複製側で破棄」「操作子は左ペイン▲▼近傍のコピペアイコン
  （2 枚重ねの書類）」。
- Part 1・3・4 は追加指示として同セッションで承認済み（Ctrl+クリック増減、ダブルクリック+Enter
  でのコメント編集、Ctrl+A のコメント包含）。

**未決事項（実装フェーズまたは将来 CR で確定）:**
- Part 5 で、複製操作の対象範囲内に閉じる依存線（複製された両端が同一複製操作内にある場合）を
  複製側でも再現するかどうか。本 CR は「分類/セクションをまたぐ」依存線の破棄のみを規定しており、
  複製内で閉じるケースの扱いは未規定。
- Part 2 の上下移動で、小分類（minor）が唯一かつ隣接 sibling がない場合のフィードバック（禁止
  カーソル表示等の UX）。
- Part 1 の Ctrl+クリック増減と既存の矩形選択のドラッグ開始点判定（アイテム上でのドラッグ開始が
  移動操作と衝突しないための当たり判定優先順位）。
- コメント編集中（Part 3）に Escape キーで編集を取り消すかどうか（Enter=確定のみ既定、Escape の
  挙動は未規定）。
