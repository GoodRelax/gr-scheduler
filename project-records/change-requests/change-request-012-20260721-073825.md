## 1. 背景・目的

CR-004..011 の実装完了後、オーケストレータが出荷版（既定起動テンプレート）に対して CR-006 で追加した
予実表示パレット・コントロール（`[A]`=Plan表示オンオフ, `[P]`=Actual表示オンオフ, `[As]`=Separate,
`[Ao]`=Overlap）をライブ検証したところ、次の不具合的挙動が判明した。

- `[A]` をオフ（Actual-only 表示のつもり）にしても、依然として 32 アイテム全件が表示され、見た目が
  変化しない。
- `[P]` をオフ（Plan-only 表示のつもり）にすると、表示アイテムが **0 件になりキャンバスが空白**になる。
- `[As]`（Separate）にすると、バー高さは半減する（18px→8px 相当）が、**下段の実績用サブレーンが
  常に空**のままである。
- `[Ao]`（Overlap）にしても、**実績バーが一切描画されない**。

**根本原因（root cause）**: CR-001 で予実データモデルを「1 アイテムが予定スパン（`startDate`/
`endDate`）と実績スパン（`actualStart`/`actualEnd`）＋進捗率（`progressRatio`）を同時に持つ」統一モデル
へ移行済みであり、パレット・コントロールの実装（`filterByPlanActualDisplay`,
`computePlanActualBars`, `src/domain/usecase/plan-actual-geometry.ts`）はこの統一モデルを正しく前提に
実装されている（読取専用で確認済み、コントロール自体に defect は無い）。

しかし、**既定起動テンプレート**（`src/app/generateTemplateDocument`, `src/app/sample-data.ts`）は
IM7 で「`old/gr-scheduler-template.json`（旧モデル）を keep-as-is で新モデルへ移植する」方針のもと
実装されており、予定と実績を **別行・別アイテム**（青 `*-Plan` ミドルカテゴリのアイテム群と、対をなす
赤 `*-Actual` ミドルカテゴリのアイテム群）として保持している。すなわち **テンプレートのどのアイテムも
`actualStart`/`actualEnd` を持たない**。赤い「実績」行は、アプリのデータモデルからは単に赤色で塗られた
通常のタスク／マイルストーンに過ぎず、予実の対応関係（同一アイテムの予定と実績のペア）としては
一切認識されない。

この結果、`filterByPlanActualDisplay`/`computePlanActualBars` は仕様通りに動作しているにもかかわらず、
「Actual-only 表示で 0 件になる」「Separate/Overlap で実績サブレーン・実績バーが常に空になる」という、
ユーザーから見て defect にしか見えない挙動が既定テンプレートに対してのみ発生する。

**本 CR は IM7 決定を supersede する。** IM7 の「予定/実績は別行のまま、青予定/赤実績、keep-as-is」
という方針を撤回し、既定テンプレートを CR-001 の統一モデルへ移行する。ユーザーは本移行方針を
2026-07-21 に承認した。

## 2. 変更内容

### Part 1 — `*-Actual` アイテムを対応する `*-Plan` アイテムへ統合

`src/app/generateTemplateDocument`（`src/app/sample-data.ts`）内の各 `*-Actual` ミドルカテゴリの
アイテムを、対応する `*-Plan` ミドルカテゴリの対のアイテムへ統合する。具体的には、実績アイテムの
`startDate`/`endDate`（マイルストーンの場合は開始日のみ）を、対応する予定アイテムの
`actualStart`/`actualEnd`（マイルストーンは `actualEnd: null`）として転記し、実績アイテムが持つ
`progressRatio` があれば同様に転記する。統合後、旧 `*-Actual` アイテムのレコード自体は削除する。

現行の対応関係（読取専用で確認済み、`src/app/sample-data.ts` の `seeds` 配列より）:

| 実績アイテム（統合元、削除対象） | 対応する予定アイテム（統合先） |
|---|---|
| `oa-ms-actual-kickoff`, `oa-ms-actual-freeze`（Milestones-Actual） | `oa-ms-plan-kickoff`, `oa-ms-plan-freeze`（Milestones-Plan） |
| `oa-phase-actual-concept`, `oa-phase-actual-dev`（Over All Schedule / Phase-Actual） | `oa-phase-plan-concept`, `oa-phase-plan-dev`（Over All Schedule / Phase-Plan） |
| `ta-phase-actual-sys1`（TeamA / Phase-Actual） | `ta-phase-plan-sys1`（TeamA / Phase-Plan） |
| `ta-swephase-actual-swe2`（TeamA / SWE-Phase-Actual） | `ta-swephase-plan-swe2`（TeamA / SWE-Phase-Plan） |

`Milestones-Plan` の `oa-ms-plan-sop`（SoP）/`oa-ms-plan-launch`（SOS）、`Phase-Plan` の
`oa-phase-plan-valid`（Validation）/`oa-phase-plan-rampup`（Ramp-Up）、`ta-phase-plan-sys2`（SYS2）/
`ta-phase-plan-sys3`（SYS3）/`ta-phase-plan-swe1`（SWE1）、`SWE-Phase-Plan` の `swe3`〜`swe6` など、
対応する実績アイテムが存在しない予定アイテムは、実績フィールドを付与しない（予定のみのアイテムとして
そのまま残す）。これにより統合後もモデルとしての疎密（一部アイテムのみ実績あり）が保たれ、
`generateSampleDocument`（ベンチマーク用フィクスチャ）が既に採っているのと同じ統一モデルの使い方
（読取専用で確認済み、`src/app/sample-data.ts` 385〜409行目付近の `showcasePath` アイテム群）に揃う。

### Part 2 — 空になった `*-Actual` ミドルカテゴリの削除

Part 1 の統合により空になる `Milestones-Actual`・`Phase-Actual`（Over All Schedule / TeamA 双方）・
`SWE-Phase-Actual` の各ミドルカテゴリは、`rebuildClassification` の再導出結果として自然に消滅する
（分類ツリーはアイテムから導出されるため、明示的な削除処理は不要）。ただし現行の 13 行（左ペインの
分類ツリー行数）は本移行により減少する見込みであり、行数・アイテム数・依存線 ID 等をアサートして
いる既存テスト（下記 §3）はこの変化に追従して更新する。

### Part 3 — 赤 `ACTUAL_FILL` の廃止・配色を彩度導出方式へ統一

現行 `src/app/sample-data.ts` はテンプレート専用の固定色 `PLAN_FILL = '#4477aa'` / `ACTUAL_FILL =
'#ee6677'` を個別に定義し、予定アイテムと実績アイテムそれぞれに直接割り当てている（読取専用で確認済み、
89〜90行目）。CR-002 で確定した方式（予実の塗り分けは、アイテムのベース色 `fillColor` 1 色から
`displayFillColor`/`actualDisplayFillColor`（`plan-actual-colors.ts`）が彩度で導出する。予定＝淡色・
実績＝濃色）に統一し、テンプレート専用の `ACTUAL_FILL` 定数と、その値を直接指定するコードは削除する。
Part 1 で統合されたアイテムは `fillColor`（ベース色）のみを持ち、実績側の濃色表示は描画時に
`actualDisplayFillColor` が導出する。

### Part 4 — `-Plan` サフィックスの整理

Part 1・Part 2 により、残存するミドルカテゴリは実質的に「予定専用」ではなく「予定と（あれば）実績を
併せ持つ」ミドルカテゴリになるため、`-Plan` サフィックスは実態と乖離し誤解を招く。次の通り改称する。

| 現行ミドルカテゴリ名 | 改称後 |
|---|---|
| `Milestones-Plan` | `Milestones` |
| `Phase-Plan`（Over All Schedule） | `Phase`（Over All Schedule） |
| `Phase-Plan`（TeamA） | `Phase`（TeamA） |
| `SYS-Phase-Plan` | `SYS-Phase` |
| `SWE-Phase-Plan` | `SWE-Phase` |
| `Integration-Plan` | `Integration` |
| `Task-Plan` | `Task` |

アイテム ID（`oa-ms-plan-kickoff` 等の `-plan-`/`-actual-` を含む識別子）は内部 ID であり画面表示には
現れないため、改称の要否・要否範囲（IDも合わせて刷新するか、後方互換のため現行 ID を維持するかを
含む）は実装フェーズで判断する（下記 §7 未決事項）。

## 6. 受入基準（Acceptance Criteria）

移行後の既定起動テンプレートに対して、次を満たすこと。

- [ ] `[A]`（Plan 表示）をオフにすると、Actual-only 表示となり、実績データを持つアイテムの実績側の
      見た目（濃色バー/マーカー）のみが表示される（現行のように「32 アイテム全件表示のまま変化
      なし」にはならない）。
- [ ] `[P]`（Actual 表示）をオフにすると、Plan-only 表示となり、予定側のみが表示される。実績データを
      持たないアイテムも予定として表示され続け、**キャンバスが空白にはならない**。
- [ ] `[Ao]`（Overlap）にすると、実績データを持つアイテムについて、実績バー（濃色）が予定バー（淡色）
      の区間に重ねて描画される。
- [ ] `[As]`（Separate）にすると、実績データを持つアイテムのレーンが予定（上段）/実績（下段）に
      分割され、**両方のサブレーンにデータが表示される**（下段が常に空、という現行の defect が
      解消される）。
- [ ] イナズマ線（進捗線）が、実績データを持つアイテム（統合後に `actualStart`/`actualEnd`/
      `progressRatio` を持つもの）に対して描画されるためのデータが存在する。
- [ ] 既存の依存線（`tpl-dep-concept-dev`, `tpl-dep-dev-valid`, `tpl-dep-sys1-sys2`,
      `tpl-dep-sys3-swe1`）が、統合後のアイテム ID に対して引き続き解決される（依存元/依存先の
      アイテムが削除されずに残っていること）。
- [ ] 品質ゲート（tsc / vitest / eslint / strictdoc / build）が引き続き green であること。

## 7. 決定事項・未決事項

**決定済み（本セッションでユーザー確定、2026-07-21）:**
- 既定起動テンプレートを、IM7 の「予定/実績を別行・別アイテムで保持する keep-as-is 方式」から、
  CR-001 で確定した統一モデル（同一アイテムの `actualStart`/`actualEnd`/`progressRatio`）へ移行する。
  本 CR は IM7 決定を supersede する。
- `*-Actual` アイテムは対応する `*-Plan` アイテムへ統合し、空になった `*-Actual` ミドルカテゴリは
  削除する。
- テンプレート専用の赤 `ACTUAL_FILL` 固定色は廃止し、CR-002 の彩度導出方式（`fillColor` 1 色から
  `displayFillColor`/`actualDisplayFillColor` が導出）に統一する。
- 残存するミドルカテゴリ名から、実態と乖離する `-Plan` サフィックスを除去する。

**未決事項（実装フェーズで確定、フォローアップとして別途判断）:**
- `[P]`（Actual 表示）をオフにした際、実績データを一切持たないアイテムまで非表示になり結果として
  Actual-only 表示が 0 件（空白キャンバス）になるケースを、明示的な空状態表示または操作抑制で
  ガードするか否か。本 CR の移行によりテンプレートの大半のアイテムは実績データを持つに至るため
  発生頻度は下がるが、ユーザーが実績データを持たないプロジェクトを新規作成した場合には引き続き
  発生し得るため、フォローアップとして別途扱う（本 CR のスコープには含めない）。
- アイテム ID（`oa-ms-plan-kickoff` 等）の `-plan-`/`-actual-` を含む命名を、ミドルカテゴリ改称
  （Part 4）に合わせて刷新するか、内部 ID として現状維持するか。
- `docs/spec/18-plan-actual.sdoc` への要求追記要否（既定テンプレートの構成は通常「実装詳細」であり
  要求文（`PLAN-L1-*`）自体の変更は不要と見込むが、実装フェーズで最終確認する）。

## 8. 段階化の注意（仕様先行・実装後追い）

CR-001 §8 / CR-002 §8 / CR-006 §8 / CR-010 §8 / CR-011 §8 と同様の理由（`docs/api/
gr-scheduler.schema.json` は F0 で SSOT 化され `tests/document-schema-conformance.test.ts` がコード
出力と突合しているため、コードより先に schema を変更するとテストが赤化する。本 CR はスキーマ変更を
伴わないが、`src/**` / `tests/**` の実装は一体で反映する方針を踏襲する）により、本フェーズでは
以下とする。

- **本フェーズで作成するのは本 CR 票のみ**とし、`docs/spec/**`・`docs/api/**`・`src/**`・
  `tests/**` は一切変更しない。
- **実装フェーズで反映する**: `src/app/sample-data.ts`（`generateTemplateDocument`）の Part 1〜4、
  および依存するテスト群（アイテム数/行数/依存線ID/ミドルカテゴリ名のアサーション更新、
  `[A]`/`[P]`/`[As]`/`[Ao]` の回帰テスト新設）。実装は一体で行い、品質ゲート（tsc/vitest/eslint/
  strictdoc/build）を緑に保つ。
- `docs/spec/18-plan-actual.sdoc` への要求追記要否は実装フェーズで判断する（上記 §7 未決事項）。
- `[P]` オフ時の空白キャンバス・ガードの要否は本 CR のスコープ外とし、フォローアップとして別途
  change-request もしくは decision で扱う。
