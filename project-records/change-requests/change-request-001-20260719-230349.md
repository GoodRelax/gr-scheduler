## 1. 背景・目的

GR JSON と MSPDI XML の相互変換を分析（フィデリティ・レポート）した結果、次が判明した。

1. **形式差（本質的損失）**: GR→MSPDI は視覚層、MSPDI→GR は工数・資源・制約層が相手側で消える（双方向・非対称）。これは根治にモデル拡張が要る。
2. **実装漏れ（修正可能）**: 両形式とも表現できるのに現 codec が写していない項目が複数ある（レポート §5）。
3. **予実モデルの方針転換**: 進捗入力／イナズマ線の検討から、予実の持ち方を**案H**へ移行する判断に至った（本会話で決定）。

本 CR は上記を1つの変更パッケージとして**別担当の実装者へ引き継ぐ**ためにまとめる。以下、Part A（予実モデル案H）・Part B（MSPDI 実装漏れ）・Part C（GRモデル追加推奨）で構成する。

## 2. 変更内容

### Part A: 予実データモデルを「案H」へ移行（最重要・requirement-change）

**方針**: 実績を「別アイテム」ではなく「**同一アイテムの実績日フィールド**」で持つ。進捗率のみ（案P）では実績の時間ズレ（着手遅れ・期間変動）を表現できず、日本の予実文化に不足するため採らない。案H は MSPDI（Baseline/Actual/PercentComplete）にも素直。

**item スキーマ変更:**
```diff
  startDate, endDate            // 予定(plan)スパンの意味に確定
+ actualStart?: IsoDate         // 実績開始（任意）
+ actualEnd?:   IsoDate | null  // 実績終了（任意, milestone は null）
  progressRatio?: number        // 進行中バーの到達割合（据え置き）
  previousPlan?                 // 変更前予定＝ベースライン履歴（据え置き）
- planActualKind                // 廃止（'plan'|'actual'）
- planGroupId                   // 廃止
```

**viewState 追加（表示スタイルは両対応・ビュー切替。実装名は Overlap / Separate を用い、`Opt1`/`Opt2` の呼称は使わない）:**
```diff
+ planActualStyle: 'overlap' | 'separate'   // default 'overlap'（重ね塗り）
  planActualDisplay: 'plan-only'|'actual-only'|'both'|'none'  // 据え置き（フィルタ）
```
- **Overlap（重ね塗り・既定）**: 予定バー上に実績/進捗を塗り重ねる。
- **Separate（分離）**: 予定バーと実績バー（`actualStart..actualEnd`）を上下に分けて描く。
- どちらも**同一データからの描画切替**であり、データ設計とは独立。**既定は Overlap**。

**イナズマ線 front の統一規則（ARCH-C-014 / progress-today-layer）:**
```
実績日あり → front = actualStart + ratio × (actualEnd − actualStart)
実績日なし → front = startDate  + ratio × (endDate − startDate)   // MSP % 相当
どちらも無 → 未着手（頂点を出さない）
```

**MSPDI マッピング（往復が素直に）:**
`startDate/endDate ↔ Task Start/Finish` ／ `actualStart/actualEnd ↔ ActualStart/ActualFinish` ／ `progressRatio ↔ PercentComplete` ／ `previousPlan ↔ BaselineStart/Finish`。

### Part B: MSPDI codec の実装漏れ（フィデリティ §5・両形式が表現可能なのに未対応）

| # | 実装漏れ | 対応方針 | 現状 |
|---|---------|---------|------|
| B-1 | 行/セクション階層 | `OutlineLevel`+`Summary` で双方向変換（`DATA-MSPDI-005`） | Export は `OutlineLevel` を 1 固定・サマリ未生成／Import は階層を読まず全タスクを `row-0` に集約 |
| B-2 | 担当 `assignee` ⇔ Resource/Assignment | Resource として出力・読取 | 双方向とも未対応 |
| B-3 | 進捗 `progressRatio` ⇔ `PercentComplete` | 数値を往復 | 双方向とも未対応 |
| B-4 | 予実/変更前予定 ⇔ `Baseline*`/`ActualStart/Finish` | Part A の actual*/previousPlan を Baseline・実績日で表現 | 未対応（Part A と一体で実装） |
| B-5 | 分割タスク `SplitParts` → マルチバー | 分割区間を GR の 1 行複数アイテムへ写す | Import が `SplitParts` を無視し連続バーに潰す |
| B-6 | 説明 `description` ⇔ `Task/Notes` | タスク単位 Notes に格納 | 未対応（Project 単位 Notes をサイドカーが占有中のため配置設計が要る） |

> B-4 は Part A のモデル変更と不可分。B-1/B-2/B-3/B-5/B-6 は Part A と独立に着手可能。

### Part C: GR モデルへの追加推奨（フィデリティ §6）

MSPDI→GR で形式差（✕）となる項目のうち、GR に取り込む価値があるもの。採否は「俯瞰日程表」という製品スコープで判断した。

| 項目 | 現状の受け皿 | 推奨 | 備考 |
|------|-------------|:---:|------|
| 担当者 `assignee` | モデルに有（配線のみ） | **採用（最優先）** | 実体は B-2 の実装作業 |
| 進捗率 `progressRatio` | モデルに有（配線のみ） | **採用** | 実体は B-3 |
| 予実・変更前予定 | モデルに有（配線のみ） | **採用** | 実体は Part A / B-4 |
| 依存の種別（FS/SS/FF/SF） | モデルに**無** | **採用（本CRに含む）** | `dependencies[]` に `linkType` 追加。MSPDI `PredecessorLink/Type`(FF=0/FS=1/SF=2/SS=3) と往復。低コスト高価値 |
| 依存のラグ/リード | モデルに**無** | **採用（本CRに含む）** | `dependencies[]` に符号付き `lagDays`（正=ラグ/負=リード）を追加。MSPDI `PredecessorLink/LinkLag`+`LagFormat` と往復。下記「ラグ／リード解説」参照 |
| 期限（目標終了日） `Deadline` | モデルに**無** | **採用（本CRに含む）** | バー端と独立の「◆ここまで」目標マーカー。`item.targetDate` 追加で表現。俯瞰日程表と相性良 |
| 制約（ルール）`Constraint` | モデルに**無** | 見送り | GR は手動配置でスケジューラが無く適用先が無い |
| 制約（ルール）`Constraint` | モデルに**無** | 見送り | GR は手動配置でスケジューラが無く適用先が無い |
| 工数/コスト/カレンダー/資源配分/タスク種別 | モデルに**無** | 見送り | 製品スコープ外（資源計画は非目的）。将来 CR で再評価 |

**依存のラグ／リード解説（`lagDays`）**: 依存線には、種別（FS/SS/FF/SF＝どの端点同士を関係づけるか）に加えて、端点間の**時間オフセット**を持たせられる。

- **ラグ（正・待ち時間）**: 先行の条件成立後に後続を**遅らせる**日数。例 `FS+10d` ＝ 先行終了の10日後に後続開始（部品調達待ち・養生・承認待ち等）。
- **リード（負・前倒し）**: 後続を先行の完了前に**重ねて**始める日数。例 `FS-5d` ＝ 先行終了の5日前に後続開始（ファストトラッキング）。
- モデルは符号付き1フィールド `dependency.lagDays`（正=ラグ／負=リード／0・未指定=なし）で表す。
- MSPDI `PredecessorLink/LinkLag`(+`LagFormat`) と往復。**注意**: MSPDI のラグは稼働時間（カレンダー）基準で単位も可変だが、GR はカレンダーを持たないため**暦日**として近似する（往復で端数がずれ得る形式差）。

## 4. 後方互換（不要）

**本製品は未公開のため、旧データとの後方互換は考慮しない。** 既存の `planActualKind`/`planGroupId` 方式のデータ（サンプル・開発中の localStorage 等）を温存する移行処理は**不要**。

- スキーマは新形状へ直接置換してよい（`schemaVersion` の引上げは任意）。
- `src/app/sample-data.ts` と各テスト・フィクスチャを案H の新形状へ更新するだけでよい。
- 旧形式を読んだ場合の挙動は未定義でよい（移行コードを書かない）。
