# DEC-001: レンダリング方式（RISK-001 対応）

- 日付: 2026-07-18
- 種別: アーキテクチャ根本判断（CLAUDE.md「重要判断の基準」: アーキテクチャの根本的選択 / リスクスコア6以上）
- 決定者: ユーザー承認
- 関連: RISK-001（score 9）, NFR-L1-002, ADR-001, ADR-005, ADR-009, ZOOM-L2-001, ARCH-C-021/033

## 背景

中規模（約50行/約1000アイテム）で 60fps のズーム/パン + 初期表示1.5秒（NFR-L1-002, Must）を、
単一HTMLの vanilla TS + SVG DOM で達成できないおそれ（RISK-001, score 9）。素朴な全ノード描画は
多数ノードの再レイアウト/再ペイントでフレーム落ちしやすい。

## 選択肢

1. **SVG + ビューポート仮想化/LOD + 早期性能PoC**（採用）
2. ハイブリッド（Canvas描画 + SVGオーバレイ/出力）
3. 全Canvas描画

## 決定

**選択肢1を採用**する。SVG を維持（SVG出力要求 IO-L1-003 と自然に整合、ヒットテスト・
アクセシビリティ WCAG 2.1 AA に有利）しつつ、以下を必須設計とする:

- ビューポート仮想化（可視領域+近傍のみ DOM 生成、画面外は非生成/遅延生成）
- 重要度 LOD 間引き（ADR-005 / ZOOM-L2-001）
- 差分描画（変更差分のみ DOM パッチ、requestAnimationFrame でバッチ更新）

**性能PoC を Phase 4 M1（ウォーキングスケルトン）の計測ゲート**とし、約1000アイテム相当の
ダミーデータで 60fps/1.5秒 の達成を先に実測する。

## 根拠

- SVG 出力（item44 / IO-L1-003）は SVG レンダリングと同一系統で自然に得られる
- WYSIWYG 編集のヒットテスト・フォーカス・スクリーンリーダー対応が SVG DOM で容易
- 仮想化により同時描画ノードは「全1000」ではなく「可視+近傍の数十〜百数十」に抑えられ、
  現規模では 60fps が現実的
- 後戻りコストが小さい（PoC で早期に方式判断できる）

## 帰結・残留リスク

- レンダラは可視判定・差分適用・LOD の複雑度を負う
- 残留リスク（RISK-001）は PoC 結果まで受容（open）。**PoC 未達の場合は Canvas 併用ハイブリッド
  等を decision として再記録し、ユーザーへ再確認**する
- ADR-009 として `30-architecture.sdoc` に明文化

## PoC 結果（2026-07-18, ユーザー立会いで実ブラウザ計測）

M1 ウォーキングスケルトン（`?bench=N`）で計測。判定は **目標規模で合格**:

| 規模 | 初期描画 | 平均FPS | p95フレーム | 実SVGノード | 判定 |
|---|---|---|---|---|---|
| 1000アイテム/50行（目標=NFR-L1-002） | 4.6 ms | 162.3 | 6.20 ms | 93 | ✅ 大差でPASS |
| 10000アイテム/50行（10倍・スコープ外） | 45.2 ms | 20.0 | 66.70 ms | 860 | ⚠️ 既知上限 |

- **DEC-001（SVG + 仮想化/LOD）は目標規模で妥当と確定**。RISK-001 は mitigated（残留: 超高密度域）。
- 方針変更（Canvas 併用等）は不要。将来の大規模要求時のみ再検討。
