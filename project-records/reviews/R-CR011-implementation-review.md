## 焦点回答: 短ビューポート時のクリッピング問題（R2/R5）

**結論: 許容できる CR 準拠の設計トレードオフ。重大度 Low（記録のみ、ブロッカーではない）。**

根拠:

1. **現実的な余裕が十分ある。** ライブ検証（1280px 幅）で en/ja とも `scrollHeight = 554px`。
   高さ 768px のラップトップでは `max-height = calc(100vh - 16px) = 752px` となり、
   554 < 752 で約 198px（26%）の余裕がある。一般的なラップトップ高（768 / 800 / 900px）は
   すべて余裕をもって収まる。クリップが顕在化するのは高さ ≲ 570px の病的に低いビューポートに限られ、
   これは本 CR が明記する対象範囲（デスクトップ利用）の外である。

2. **clamp フォント床は幅方向の縮退を担う。** `font-size: clamp(11px, 0.95vw, 13px)` は
   幅 1368px で 13px 天井、幅 ≲ 1158px で 11px 床に達する。狭幅時はフォントで吸収するため
   横方向の破綻はない。ただしフォント縮退は**高さ**を直接は保証しない（狭幅では列が細くなり
   本文が縦に伸びうる）点は本質的な限界である。

3. **CR 準拠の代替手段が存在しない。** 高さが真に不足した場合、取りうる選択肢は
   (a) スクロール = Part 1 で禁止、(b) 段組み崩し = Part 2 で禁止、(c) 床未満へのフォント縮小 = Part 4「少しだけ」に反する、
   (d) `overflow: hidden` によるクリップ、の 4 択であり、CR の順序規定に照らすと (d) が残余として選ばれる。
   したがってクリップは**設計の欠陥ではなく、CR の「スクロール禁止・フォント微縮は最終手段」という
   明示要求から不可避に導かれる帰結**である。実装 JSDoc（L283-301）にこの戦略が明文化されている点も適切。

**留意（spec レベル・ユーザー判断事項）:** `overflow: hidden` は、内容が収まる限りにおいて
`overflow: auto` と挙動が同一（スクロールバー非表示）でありながら、真に収まらない病的ケースでは
`hidden`（=内容が到達不能に消失）ではなく `auto`（=スクロールで到達可能）に「優雅に劣化」する。
Part 1 が禁じるのは「スクロールを収まり機構として設計に織り込むこと」であって、
デスクトップでは決して発火しない安全網としての `auto` はその趣旨に反しないとも解釈しうる。
本実装は CR 本文および不変条件テストが `overflow: hidden` / `!= auto` を明示的に固定しているため
letter どおり `hidden` を採用しており妥当だが、「クリップ（内容消失）」対「スクロール（到達可能）」の
残余トレードオフは仕様上の緊張点として記録し、必要なら change-manager 経由でユーザーに提示するのが望ましい（Low-1）。

## 指摘一覧

### Low-1 [R2/R5] 病的短ビューポートでの内容クリップ（到達不能）
- **箇所**: `.grsch-help-dialog { max-height: calc(100vh - 16px); overflow: hidden; }`（L318-319）
- **問題**: 高さ ≲ 570px のビューポートで内容が `max-height` を超えた場合、スクロールではなくクリップされ、
  末尾の内容が到達不能になる。
- **影響**: デスクトップ主対象では非顕在（768px 高で 198px の余裕）。対象範囲外の低ビューポートに限定。
- **修正案（任意・spec 判断）**: (a) 対応最小ビューポート高をドキュメント化する、
  または (b) Part 1 の趣旨解釈をユーザーに確認し、非発火の `overflow-y: auto` を安全網として採用するかを検討する。
  いずれもコード欠陥ではないため本フェーズでの修正は必須ではない。

### Low-2 [R3] ファイル冒頭 JSDoc の記述ドリフト（"two-column" が実態と不一致）
- **箇所**: `help-modal.ts` L6 `"...ALL app features on one screen in a two-column layout..."`
- **問題**: 現行実装は `column-count: 3`（3 段組み）であり、CR-011 の主眼そのものが 3 段組み維持である。
  冒頭 JSDoc の "two-column" は旧仕様のまま残っており実態と矛盾する（命名・記述の正確性 = item60 隣接）。
- **影響**: 動作影響なし。将来の読み手を誤誘導する保守性上の軽微な問題。
- **修正案**: L6 の "two-column layout" を "three-column layout" 等へ更新（`HELP_MODAL_STYLESHEET` の
  JSDoc L283-301 は正しく 3 列を記述しているため冒頭のみ）。
